# 人机协作报告

## 1. AI 交互日志

### M1 数据处理

在数据处理模块中，我先让 AI 帮我梳理纽约黄出租车 2023 年 1 月数据适合保留的核心字段，并根据课程要求设计数据质量报告内容。AI 给出的建议包括统计缺失率、重复值、异常值数量，以及对时间字段和费用字段进行逻辑校验。我采纳了这部分建议，并进一步要求在代码注释中写明每一步清洗策略的理由，例如删除 `trip_distance <= 0` 的记录、删除上车时间晚于下车时间的记录、保留 `passenger_count` 在合理范围内的样本等。

AI 在早期建议中没有特别强调“只读取必要列”，我在实际运行时发现原始数据有三百多万条，全量读取会增加内存压力，因此我补充了 `get_required_columns()` 方案，只在主流程中读取后续分析和建模需要的字段。这个修改来自我在真实运行时对性能问题的观察。

### M2 分析可视化

在分析可视化模块中，我让 AI 根据题目要求拆解四类分析任务，并设计每项至少一张图。AI 给出的结构包括：

- 分小时需求折线图
- 工作日/周末对比图
- 上下车区域 TOP10 柱状图
- 区域小时热力图
- 行程距离与车费散点图
- 自选拥堵分析图

这部分整体建议比较可靠，我大多采纳了。不过在实际生成 seaborn 图表时，AI 最初使用了 `palette` 但没有设置 `hue`，运行时出现了未来版本警告。我根据警告信息对代码做了修改，改为使用 `color` 固定颜色，避免未来版本兼容问题。这是一次典型的“AI 能快速成型，但细节需要人工验收”的案例。

### M3 预测模型

在预测模块中，我先让 AI 帮我明确“需求预测”的建模口径。AI 建议不要直接预测单条订单，而要把任务转换为“区域 + 小时”的聚合需求预测。我采纳了这个思路，并用 `pickup_date + pickup_hour + PULocationID` 聚合样本，构建 `demand_count` 作为标签，再加入 `avg_distance`、`avg_fare`、`avg_duration`、`avg_speed`、`is_peak`、`is_weekend` 和 `prev_hour_demand` 等特征。

在实现过程中，AI 一度遗漏了一个现实问题：原始数据中混入了极少数不是 2023 年 1 月的异常日期记录。我在抽样查看聚合结果时发现了这一点，因此在 `prepare_demand_dataset()` 中补充了 `pickup_date` 前缀过滤，只保留 `2023-01` 的样本，以保证实验口径一致。

另外，AI 实现神经网络时依赖 `torch`，而我本地最初把依赖错装到了 `base` 环境，导致运行时报 `ModuleNotFoundError: No module named 'torch'`。这个问题不是代码逻辑错误，而是环境管理问题。我通过重新确认解释器路径、修复 `conda` 基础环境、再将依赖安装进 `work` 环境后解决了这个问题。

### M4 问答接口

在问答系统部分，我首先要求 AI 不要直接调用大模型，而是先做规则匹配型自然语言问答，以保证基础功能稳健可控。AI 设计了基于关键词与正则表达式的路由方案，可以识别至少五类问题：

- 时段需求查询
- 工作日/周末对比
- 热门区域查询
- 车费关系分析
- 需求预测
- 高峰特征查询

这一方案与课程要求较吻合，我基本采纳。但 AI 最初生成的占位版本只有一条“待实现”输出，因此我继续要求它补上参数提取函数 `_extract_hour()` 和 `_extract_zone_id()`，并为每类问题返回“数字结论 + 图表路径”。这个阶段体现出 AI 在搭骨架方面效率很高，但如果不继续追问，它可能会停留在表面完成状态。

在后续优化阶段，我又引入了 Qwen 作为兜底问答方案。规则系统负责回答可结构化的问题，例如“18点需求多少”“区域132在18点需求预测”“热门区域有哪些”；当问题超出规则覆盖范围时，再调用 Qwen 给出解释性回复。这样做的原因是：规则系统在准确性上更强，Qwen 在自然语言表达和开放式解释上更强，两者结合后，既满足作业对“自然语言问答”的要求，也能体现更高的系统完整度。

## 2. 三阶段对比

这里选取“问答系统的时段需求查询功能”作为典型功能，对比 Native 版、Prompt 版和 Vibe 版三种开发方式。

### Native 版（独立写）

如果完全由我自己手写，我通常会先写一个死板的命令行循环，再通过 `if-else` 判断问题中是否包含“几点”和“需求”两个词，手动拆字符串得到小时数，然后再到数据里统计对应时段订单量。这种方式的优点是我对每行代码的含义都最清楚，也最容易控制细节；缺点是开发速度偏慢，尤其是在整个作业模块较多时，我容易把精力过度消耗在样板代码上。

### Prompt 版（AI 辅助）

在 Prompt 版里，我先明确告诉 AI：问答系统至少支持五类问题，先从规则匹配做起，再逐类实现处理函数。AI 很快给出了 `_extract_hour()`、问题分类、以及 `answer_hourly_demand()` 这类函数框架。我再根据自己的理解调整细节，例如要求返回“数字结论 + 图表路径”，并补充更符合课程要求的中文输出。这个版本的效率明显更高，而且我仍然保留了对核心逻辑的把控。

### Vibe 版（对话驱动）

Vibe 版更像是“先说出我要什么效果，再逐步让 AI 往那个方向逼近”。例如我先说“我需要命令行问答循环，用户输入自然语言问题，系统匹配功能并返回结果”，AI 会先产出一个可跑的骨架；接着我再根据运行效果说“这里要支持区域预测”“这里要加正则提取小时”“这里返回路径而不是只返回数字”。这种模式上手最快，尤其适合从 0 到 1 搭原型，但也最需要持续验收。如果我不主动检查，AI 可能会留下占位代码、编码问题或规则覆盖不完整的问题。

在加入 Qwen 的阶段，这种对话驱动方式也很明显：我先提出“希望完整度最高”，AI 并没有只停留在把接口连上，而是进一步建议采用“规则优先、Qwen 兜底”的架构，并通过环境变量控制 API Key，使系统在没有联网能力时仍然可运行。这种改进比单纯把大模型直接塞进问答循环更适合作业场景。

### 效率与理解深度分析

从效率上看，Vibe 版最快，Prompt 版次之，Native 版最慢；但从理解深度上看，Native 版最高，Prompt 版居中，Vibe 版最低。真正适合本次课程作业的方式，其实是“Prompt 版 + 必要的 Native 验证”：让 AI 快速给出结构与初稿，再由我根据课程要求、运行结果和异常情况去修正。这种混合方式兼顾了开发效率和对代码的理解深度，也更符合“人机协作编程”的教学目标。

## 3. 反思

完成这次作业后，我对 AI 工具的能力边界有了更清晰的认识。首先，AI 非常擅长把模糊需求快速结构化，尤其适合帮助我拆分任务、搭建项目骨架、生成重复性代码，以及给出一套看起来完整的实现思路。在本次作业中，从目录设计、模块划分，到质量报告、图表分析、建模接口和问答系统，AI 都显著提升了起步速度。

但与此同时，我也更明确地感受到，AI 生成的“完整”并不等于“正确”或“可提交”。例如图表代码中的警告、主文件中的中文乱码、依赖安装到错误环境、异常日期记录混入 2023 年 1 月数据、问答系统最初只是占位实现等问题，都说明 AI 往往优先满足表面结构，而不是自动保证工程细节万无一失。真正决定项目质量的，仍然是人是否进行验证、质疑和修正。

我认为 AI 当前最适合扮演的是“高效搭档”而不是“全自动替代者”。它能在短时间内给出备选方案、加快编码速度、减少样板劳动，但在高质量作业或真实项目中，人必须负责边界判断、实验口径统一、异常结果解释、环境排查与最终验收。换句话说，AI 的强项在于加速，而人的价值在于校准。在今后的学习中，我会更主动地把 AI 用在“生成初稿、拓展思路、比较方案”上，而把“验证正确性、理解原理、保证交付质量”牢牢掌握在自己手里。

## 4. Qwen System Prompt 设计与迭代

为了满足选做项“接入大模型 API，并展示 System Prompt 设计与迭代过程”，我采用了 Qwen 的 OpenAI 兼容接口，并将其作为规则问答失败后的解释性兜底层。

### 初版 Prompt

初版 Prompt 只要求模型“作为数据问答助手，用中文回答用户问题”。这个版本的问题在于约束太弱，模型有可能根据常识自由发挥，导致回答脱离本地分析结果。

### 第二版 Prompt

第二版 Prompt 加入了三条关键约束：

1. 只能基于本地已计算出的统计结果、图表和模型指标回答。
2. 不允许编造数据或图表路径。
3. 遇到超出支持范围的问题，要明确说明“本地规则系统暂未直接支持”。

这一版本显著降低了幻觉风险，但回答风格偏保守，有时不够像“解释性回复”。

### 最终版 Prompt

最终版在第二版基础上，增加了本地上下文摘要，例如：

- 清洗后数据量
- 图表数量
- 热门上车区域前 3
- 平均车费
- 平均行程距离
- 随机森林与神经网络的真实 MAE / RMSE

这样做的好处是：即使用户的问题没有被规则系统直接命中，Qwen 也能在真实上下文之内生成更具体、更自然的解释，而不是给出泛泛而谈的空洞回答。

### 设计结论

通过这次迭代，我认识到大模型接入并不是“只要能调 API 就够了”，真正决定效果的是提示词是否把模型的回答边界限定在“本地事实”内。对于数据问答类系统，Prompt 的核心不是追求更华丽的表达，而是尽量减少幻觉、增加可解释性，并让模型与本地规则系统形成互补关系。
